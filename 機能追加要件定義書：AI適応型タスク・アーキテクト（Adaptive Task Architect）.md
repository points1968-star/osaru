# **機能追加要件定義書：AI適応型タスク・アーキテクト（Adaptive Task Architect）**

**ドキュメントバージョン:** v1.0 **作成日:** 2026-05-09 **対象プロダクト:** タスク管理アプリ **ステータス:** Draft

## **1\. 概要**

### **1.1 背景と目的**

既存のタスク管理における「大きなタスクへの着手抵抗」「放置タスクによる心理的負債」「完了後の達成感の欠如」という3つの心理的障壁を排除する。AI秘書「アン」が、ユーザーの思考を調律（チューニング）し、タスクを実行可能な「資産」へと変えることを目的とする。

### **1.2 解決する課題**

| 課題 | 現状 | 本機能後 |
| :---- | :---- | :---- |
| **実行の遅れ** | タスクが巨大で何から手をつけるべきか不明 | AIが「最初の一歩」と工程を自動提示 |
| **心理的負債** | 溜まった未完了タスクがストレスになる | AIが「外科的処置」を提案しリストを浄化 |
| **継続意欲の減退** | 完了しても「こなした」だけで充実感がない | AIが仕事の価値を言語化し自己効力感を向上 |

### **1.3 スコープ**

* **対象（In Scope）**  
  * 入力タスクのAI自動分解（マイクロ・アクション変換）  
  * 停滞タスクの自動検知とAI介入（外科的処置）  
  * 完了実績のAI要約・価値言語化（振り返り）  
  * AI秘書「アン」のキャラクター・トーン管理  
* **対象外（Out of Scope）**  
  * カレンダー自動予約（次フェーズ）  
  * 外部サービス（Slack等）との自動連携

## **2\. ユーザーストーリー**

**As a ユーザー**

I want to 秘書に相談するようにタスクを整理・評価してもらいたい

So that 迷いなく着手でき、毎日を達成感を持って終えることができる

### **2.1 主要シナリオ**

* **シナリオA：分解（Decomposition）**  
  ユーザーが「安全パトロールの改善報告」を入力。AIが即座に「最初に行う5分間の作業」と4つの工程を提示する。  
* **シナリオB：介入（Intervention）**  
  3日間動いていないタスクに対し、AIが「このタスクは一度凍結しませんか？」または「さらに細かく分けましょうか？」と提案する。  
* **シナリオC：振り返り（Reflection）**  
  1日の終わりに「本日の桂様の実績」を表示。単なるリストではなく、その仕事がプロジェクトに与えた好影響を解説する。

## **3\. 機能要件**

### **3.1 分解・整理機能（Phase: Decomposition）**

* **FR-01：マイクロ・アクション変換**  
  * タスク入力時、AIが「最初の一歩（2分以内の動作）」を特定する。  
  * 3〜5つのステップに構造化して表示する。  
* **FR-02：知的助言の付与**  
  * タスクの内容に基づき、効率化のコツや注意点を「アン」の口調で一言添える。

### **3.2 外科的処置機能（Phase: Surgical Intervention）**

* **FR-03：停滞タスクの自動検知**  
  * updated\_atから72時間（設定変更可）以上変化がないタスクを「停滞」と判定する。  
* **FR-04：介入アクションの提示**  
  * 停滞タスクに対し、AIが「再分解」「一時凍結（Maybeリストへ移動）」「削除（目的喪失と判断）」の3つの選択肢をユーザーに提示する。

### **3.3 振り返り機能（Phase: Reflection）**

* **FR-05：バリュー・ナラティブ生成**  
  * 完了済みタスクの内容を分析し、その社会的・組織的価値（例：リスク回避、工期短縮への寄与）を言語化する。  
* **FR-06：デイリー・サマリー**  
  * 1日の終わりに「アン」からの労いと、翌日への布石となるアドバイスを提示する。

## **4\. 非機能要件**

* **4.1 パフォーマンス**: AIによる分解・提案生成は4秒以内に完了すること。  
* **4.2 精度の定義**: 「アン」のキャラクター性（知的・理知的・丁寧）を損なわないプロンプト管理を行う。  
* **4.3 データの整合性**: AIの提案を承認した場合のみ、既存のタスクリスト（DB）に反映させる。

## **5\. UI/UX要件**

* **5.1 介入UI**: 停滞タスクはリスト上でカードの端がわずかに「色褪せる（グレーアウト）」などの視覚効果を与える。  
* **5.2 ダイアログ形式**: AIとの対話は、押し付けがましくない「提案型チャットバブル」を採用する。

## **6\. システム要件・技術要件**

### **6.1 AIレスポンスのJSONスキーマ（例）**

JSON

{  
  "phase": "decomposition",  
  "first\_step": "現在の報告書雛形を1分で開く",  
  "steps": \[  
    "指摘箇所の写真をフォルダに集める",  
    "各写真に対する是正内容をメモする",  
    "是正前後の比較表を作成する"  
  \],

  "value\_summary": "このタスクの完了は、現場の安全基準を一段階引き上げ、リスク回避に直結します。"  
}

## **7\. 受け入れ条件（Acceptance Criteria）**

* **AC-01**: ユーザーがタスクを入力後、自動的に「最初の一歩」を含む分解案が表示されること。  
* **AC-02**: 3日放置されたタスクに「外科的処置」の提案アイコンが表示されること。  
* **AC-03**: 完了タスクが存在する場合、1日の終わりに「価値の言語化」を含んだ日報が生成されること。
